# 电赛白皮书界面设计系统

这是整个应用的视觉与交互单一事实来源。页面可以有不同结构，但必须共用同一套字体、色彩、动效节奏和交互语法。

## 设计命题

界面是一册带个人署名、可以立刻检索的数字故障手册，不是后台管理系统，也不是由表单和待办堆成的工作台。

- 首页先出现“人”：未登录时是登录 / 注册入口，登录后是头像、名字、贡献与收藏。
- 搜索像频段：静止时只有一条干净的输入轨道，聚焦后才出现扫描、建议和明确动作。
- 内容像纸：下滑自然进入连续文档索引，阅读面稳定、不透明，适合长时间使用。
- 导航像卫星：位于页面边缘，需要时围绕当前页面展开，不占据整条顶栏。
- 动效有因果：只解释“身份出现、搜索被激活、内容可进入”。

首页借鉴 [`autopoet/my-blog`](https://github.com/autopoet/my-blog) 及其实际依赖的 `valaxy-theme-yun`：身份锚点、淡色雾面背景、衬线标题、细线分节、长留白、纵向入场线和向下内容流。参考源码中的主色是 `#0078E7`，主体文字是中性黑灰；不把社交图标里的紫色误当成全站主题色，也不复制头像、文案或组件代码。电赛白皮书在这套方法里保留真实搜索、协作身份和审核状态。

## 受众、任务与语气

- 受众：电子设计竞赛学生、知识贡献者、审核员。
- 核心任务：从可观察的故障现象出发，快速找到下一步可执行检查。
- 语气：技术、编辑感、有探索性；不做营销式热闹，也不做静止的目录页。

## 结构家族

- 首页：`Identity Scroll / Index-First`。首屏围绕身份与核心搜索建立关系，下滑进入连续故障文档。
- 查 BUG：`Index-First`。搜索结果是高密度文档索引，不做筛选卡片墙。
- 文档：`Long Document`。正文是稳定纸面，目录、编辑、评论与版本信息围绕正文工作。
- 提交与审核：`Workbench`。Diff、版本责任人与审核动作处于同一工作面。
- 登录与个人页：沿用身份锚点、细线标签页和纸面文档流，不创造第二套系统。

## 视觉主题

所有实现值位于 `frontend/tokens.css`，组件只能引用 token。

- 底色：接近纸白的冷蓝灰；正文承载面可以使用纯白。
- 氛围：首页背景只允许极低对比的淡蓝、淡青和极弱粉雾与网格；它只能出现在背景，不能成为卡片填充。
- 墨色：冷黑与深灰，正文不使用低对比灰。
- 强调色：来源明确的信号蓝 `#0078E7`，视口占比不超过 5%；暖橙只可作为收藏状态的小标记。
- 状态色：绿色只代表审核通过，红色只代表删除或错误，不能承担装饰。
- 禁止玻璃模糊、彩色光球、大片渐变填充和纯装饰卡片。

## 字体

- 展示：`--font-display`，中文屏幕宋体序列，字重 700，正体；只用于品牌、身份名和主要文档标题。
- 正文与界面：`--font-body`，中文黑体序列，字重 400。
- 测量值、序号、版本：`--font-outlier`，等宽字体；只用于数据角色。
- 搜索、按钮、表单和正文不使用宋体；标题尺寸克制，不以超大字号制造“高级感”。
- 标题与正文必须形成明确的字体角色差；同一页面最多三套字体。
- 正文行长控制在 45–75 个中文字符的可读范围内。

## 导航

- 页面切换使用一颗不透明悬浮球：当前页面是核心球，点击、悬停或键盘聚焦后，其余入口沿弧线展开。
- 桌面端悬浮球位于左下；移动端位于底部中央并向上展开，避开安全区。
- 球体使用单一工程蓝和纸面卫星节点，不使用玻璃模糊或彩色光晕。
- 品牌固定在左上，仅包含图标和名称，不绘制整条顶栏。
- 全局搜索位于右上，只在当前页面没有主搜索时出现。
- 登录页只保留品牌。

## 搜索

- 首页搜索是页面的“故障频段”：默认由上下细线、搜索图标、输入文字和一个圆形动作组成，不绘制常见圆角白框。
- 聚焦时扫描线只经过一次，图标从静止变为定位状态，动作圆点向前推进；焦点环立即出现，不参与动画。
- 联想结果是唯一允许悬浮的纸面层，使用细边与单层阴影。
- 首页和查 BUG 页各有一个主搜索；全局搜索必须隐藏，避免重复。搜索占位文案只说一次，不增加提示标签和示例副标题。

## 动效

每页最多三种原语：

1. 身份出现：头像 / 标记、名字和动作在 500ms 内完成一次编排入场。
2. 搜索激活：聚焦时扫描线单次经过，结果纸层随后出现；不无限循环。
3. 当前项反馈：文档行的箭头或细线移动 1–4px；不同时叠加缩放、旋转、阴影和变色。

只动画 `transform` 与 `opacity`。焦点环立即出现，不参与动画。`prefers-reduced-motion: reduce` 下取消空间位移，反馈不超过 150ms。

## 页面规则

### 首页

- 首屏不得出现营销副标题、功能卡片或待办式统计面板。
- 未登录时身份区只提供“登录”“注册”；登录后显示真实用户名、真实贡献数和真实收藏数。
- 主动作只有描述故障并搜索；搜索下方不重复同一句提示。
- 下滑后直接展示真实故障条目，条目总数与内容只能来自接口。
- 贡献与收藏必须跳转到个人页对应标签；文档收藏必须有真实数据闭环。

### 查 BUG

- 不使用卡片墙，不增加未实现筛选。
- 每条文档显示序号、标题、现象摘要和真实 ID。
- 文档行锚定在一条排障轨迹上；交互只突出当前行，不遮挡其他结果。

### 文档

- 桌面为目录、正文、阅读进度；移动端为单列。
- 正文使用稳定不透明纸面，行内编辑发生在原位置。
- 评论默认收起；选中文字后才能出现评论动作。
- 版本必须显示修改者、审核者和日期。

### 审核与版本

- 审核页先显示责任信息，再显示字段级 Diff。
- 新增与删除必须同时使用符号、文字和颜色表达。
- 待审核版本不能覆盖公开正文。

## 可访问性与响应式

- 必须检查 `320 / 375 / 414 / 768 / 1280 / 1440px`。
- `html` 与 `body` 使用 `overflow-x: clip`，不得用横向隐藏掩盖错误。
- 所有触控目标至少 `44×44px`，可点击文字不可换成两行。
- 每个悬停反馈都有 `:focus-visible` 或触控等价；焦点环对比度至少 3:1。
- 装饰线与节点使用 `aria-hidden`，图标按钮提供可读名称。

## 工程约束

- 使用语义 HTML、CSS Modules 和少量 React 状态，不引入只为动效服务的库。
- 不改动搜索、登录、编辑、审核等已有业务契约。
- 不写组件内临时颜色或字体；缺少值时先补 token。
- 功能不能回答用户任务时就删除，不用副标题、徽章或卡片填空。
- 前端改动至少通过 `npm run lint` 与 `npm run build`。
